## Introduction

Deep learning is extraordinarily effective across vision, language, and audio. In some sense, we feel like we have the ultimate learning algorithm. But step outside these standard domains, and there are plenty of tasks we completely ignore. Learning the primality function. Learning a boolean circuit or a parity function. Deciding whether a graph is connected. One can cast these as supervised learning problems and throw a transformer at them. In practice, training will fit the data perfectly, yet fail to generalize at all.

But is there a way around it? The answer is yes — with a caveat. These tasks admit short programs: a few lines of Python. So instead of training a model to mimic the function, we can search for the exact program that fits the data. This is empirical risk minimization (ERM) over a discrete program class, and classical learning theory tells us this approach generalizes from remarkably little data. 

Awesome? What's the catch. The catch is the search. Exhuastive search over the set of all Python functions, one by one, is infeasible. There is an exponential (in the target program length) amount of python program to enumerate and it will take forever to find an actual program that fits the data.

So can we find an algorithm that is both sample- and compute-efficient? The answer is a humble NO. In general, it is impossible to guarantee both simultaneously — there are fundamental computational-statistical tradeoffs. But we can use heuristics. The idea behind LLM-PV is to use a pretrained LLM as a search prior: it won't always work, but when the LLM places enough mass on good programs, ERM-style selection becomes tractable without exhaustive enumeration.

<div class="key-message">
  Pretrained LLMs can act as search priors for empirical risk minimization over short programs.
  This makes classical program-based ERM practically searchable while preserving its strong sample-efficiency advantages.
</div>

The story has four parts:

1. **Deep learning fails on program-like tasks**: SGD-trained networks fit the training data but often fail to recover the underlying rule.
2. **ERM over short programs is sample-efficient**: PAC-style bounds scale with program length, not ambient input dimension.
3. **Brute-force ERM is computationally infeasible**: naive enumeration over programs is exponential in description length.
4. **LLM-PV bridges the gap**: a pretrained LLM provides a structured proposal distribution over candidate programs, making ERM-style selection feasible in practice.

## Part I: Deep learning fails on algorithmic tasks

### Where our usual intuition comes from

Most intuitions about sample efficiency come from benchmark regimes where deep learning is well aligned with the problem structure. In images, language, and speech, a sufficiently expressive model trained with SGD often generalizes from finite data in a stable and predictable way.

Algorithmic tasks are different. Examples include:

- **Parity:** decide whether $\langle s, x \rangle \bmod 2 = 1$ for an unknown sparse vector $s$.
- **Primality:** decide whether a large integer is prime.
- **Pattern matching:** determine whether a fixed motif appears in a binary string.
- **Balanced parentheses:** decide whether a bracket string is a valid Dyck-2 word.

These are natural computational problems with compact exact implementations. But when we train large neural models on labeled examples from such tasks, a consistent failure mode appears.

### The failure mode

Across several 1B-scale language models, training on a few hundred labeled examples produces near-perfect training accuracy on tasks such as parity, Dyck-2, and cellular automata parity. Test accuracy, however, stays near chance. This behavior persists even when scaling training data dramatically.

This is not just a matter of architecture choice, hyperparameter tuning, or lack of pretraining. The pattern remains across multiple backbones, wide sweeps of learning rate and batch size, and even strong in-context or fine-tuning baselines.

### Why SGD struggles

The statistical query perspective offers a clean explanation. Gradient-based methods access the data distribution through aggregated statistics, and for families such as sparse parity, this creates an inherent bottleneck. Informally, the target rule is simple as a program but difficult to identify through gradient-based correlation estimates.

For planted $k$-parity over $n$ variables, finite-precision mini-batch coordinate SGD with batch size $B$ and $b$-bit quantization requires at least

$$
q = \Omega\left(\frac{\sqrt{\binom{n}{k}}}{2^b}\right)
$$

fresh examples to achieve nontrivial population error.

So the issue is not that the target rule is intrinsically complex. The issue is that **SGD is the wrong search procedure for this hypothesis structure**.

## Part II: ERM over short programs is sample-efficient

### Think of the target as code

Suppose the target rule can be expressed as a program of length $L$ over an alphabet $\Sigma$. Then the relevant hypothesis class is the set of valid programs of length at most $L$, and its size is bounded by

$$
|\mathcal{H}_L| \le |\Sigma|^L.
$$

This immediately gives a classical PAC-style guarantee. If the shortest consistent program has length $L$, then with probability at least $1-\delta$ over the sample,

$$
\mathrm{test\ error}
\le
\frac{L \log |\Sigma| + \log(2L^2/\delta)}{m}.
$$

The key point is that the sample complexity depends on **description length**, not on raw input dimension. A compact parity program can therefore generalize from a surprisingly small number of examples.

### Length-first ERM

The most direct implementation is length-first enumeration: search over all valid programs of length 1, then 2, then 3, and so on, and return the first one that fits the data.

Conceptually, this is beautiful. It is simple, exact, and inherits the sample-efficiency guarantees of finite-class ERM.

But computationally it is hopeless. With $|\Sigma| = 128$ and moderate program length $L$, the number of candidates scales like $|\Sigma|^L$, which becomes enormous very quickly.

### The central tension

This yields a sharp tension between statistical and computational efficiency:

| | **Online SGD** | **Length-First ERM** |
|---|---|---|
| **Samples for low error** | $\Omega(n^{k/2})$ | $O(k \log n)$ |
| **Per-sample compute** | $O(1)$ | $\Omega(n^{\Theta(k)})$ |

SGD is computationally cheap but statistically wasteful. Program ERM is statistically efficient but computationally intractable.

So the question is: **can we keep ERM's sample efficiency without paying for uniform search?**

## Part III: LLM-PV makes ERM searchable

### From uniform search to proposal priors

The real bottleneck in program ERM is not verification. Given a candidate program, checking whether it fits the data is easy. The bottleneck is proposing promising candidates in the first place.

Uniform search over programs of length $L$ assigns probability about $|\Sigma|^{-L}$ to any fixed target. That is why brute-force ERM is effectively impossible.

A pretrained LLM changes this proposal distribution. Conditioned on labeled examples, it does not generate code uniformly from program space. Instead, it concentrates probability mass on short, structured, plausible rules suggested by the data.

That is exactly the missing ingredient.

### The LLM-PV procedure

LLM-PV, or propose-and-verify, is simple:

1. **Construct a prompt** from the training examples $S_{\mathrm{tr}}$.
2. **Sample** $k$ candidate programs from the LLM.
3. **Compile and execute** each candidate in a sandbox.
4. **Select** the program with lowest validation error on $S_{\mathrm{val}}$.

No gradient updates are used. The LLM does not perform the learning objective itself. It only proposes candidate hypotheses. Final selection is still performed by held-out empirical error, just as in ERM.

### The formal guarante

The guarantee separates the statistical role of validation from the search role of the LLM. Let $p_\epsilon(S_{\mathrm{tr}})$ denote the probability that a fresh LLM proposal has population error at most $\epsilon$. Then after $k$ trials, with probability at least $1-\delta$,

$$
\mathrm{err}_D(h^\star)
\le
\epsilon + 2\sqrt{\frac{\log(4k/\delta)}{2m_{\mathrm{val}}}}.
$$

The LLM enters only through $p_\epsilon$: the chance that it proposes a good program. It does not need to be trusted as a final predictor. It only needs to place enough mass on good hypotheses so that one appears within a modest number of trials.

This replaces the effective search size $|\Sigma|^L$ with roughly $1/p_\epsilon(S_{\mathrm{tr}})$. Whenever the LLM is much better than uniform search, the search becomes dramatically cheaper.

### More data does not rescue SGD

One natural objection is that neural baselines simply need more examples. But this does not seem to be the issue. Even when trained on up to 100,000 examples, SGD-trained models can still remain near chance on parity-style tasks while fitting the training data perfectly.

Because the output is executable code and the search trace is logged, LLM-PV is interpretable in a concrete way. The learned hypothesis can be read, tested, modified, and verified directly.

LLM-PV is not useful only on synthetic program-learning benchmarks. On standard tabular datasets with only 100 training samples, it is competitive with strong classical and foundation-model baselines.

So the propose-and-verify view is broader than exact program recovery. It can also be competitive in low-data structured prediction settings where the hypothesis class is not purely algorithmic.

## What this does and does not say

This argument does **not** claim that LLM-PV replaces deep learning on standard perception or language tasks. In those domains, the standard deep-learning pipeline remains extremely effective.

What it *does* show is that **classical ERM over programs becomes practically relevant again** when a pretrained LLM provides a strong search prior. The LLM does not replace verification, and it does not change the learning objective. It changes only the proposal mechanism.

This also explains the SHA-256 result. If the target does not admit a short exploitable program, then search bias cannot help. In that case, failure is not a weakness of LLM-PV in particular. It reflects a genuine absence of compressible structure.

More broadly, the method highlights a useful division of labor:

- the **LLM proposes**
- the **data verifies**
- the **selection rule remains ERM**

That separation is what makes the approach both effective and auditable.

  <div class="takeaway-label">Takeaway</div>
  <p style="margin-bottom: 1rem;">Pretrained LLMs can make program-based ERM practical by turning brute-force search into guided search. The mechanism has four parts:</p>
  <ol>
    <li><strong>Short programs generalize.</strong> If the target has a length-$L$ description, ERM needs only $O(L \log |\Sigma|)$ samples.</li>
    <li><strong>SGD can still fail badly.</strong> On SQ-hard families such as parity, gradient-based learners may require exponentially many samples despite the target being simple as code.</li>
    <li><strong>Brute-force ERM is computationally intractable.</strong> Uniform search over programs scales like $|\Sigma|^L$.</li>
    <li><strong>LLM-PV closes the gap.</strong> A pretrained LLM proposes plausible candidate programs, and held-out verification selects the best one, preserving the spirit of ERM while making the search feasible.</li>
  </ol>
  <p style="margin-top: 1rem; margin-bottom: 0;">The result is a learning paradigm that can recover exact algorithmic rules from a handful of examples, generalize beyond the training distribution, and produce executable hypotheses that can be read and verified directly.</p>
